# .claude-memory/

Per-machine, per-workspace Claude Code auto-memory pools. **Git-synced
across machines** so a memory written in any session lands here and
flows to every other machine you run OPS on. This is one of the
load-bearing ideas in OPS: durable knowledge lives in tracked files on
disk, not in a context window that dies on compaction.

## How it's wired

Claude Code writes auto-memory to
`~/.claude/projects/<encoded-cwd>/memory/`, where `<encoded-cwd>` is the
working directory with every `/` replaced by `-` (e.g., `~/OPS` →
`-home-operator-OPS`, a root session → `-root`). Left alone, that memory
would sit under `~/.claude/`, untracked and stranded on one machine.

`WORKFORCE/bin/ac-memory-init` walks every encoded subdir Claude Code
has created under `~/.claude/projects/`, moves each `memory/` dir into a
git-tracked pool, and symlinks the original path to it:

```
~/.claude/projects/<encoded-cwd>/memory/
    → symlink → ~/OPS/.claude-memory/<hostname>-<encoded-cwd>/
```

Everything Claude writes to memory now lands directly in this
git-tracked directory. `ac-memory-init` is idempotent — safe to re-run
on a machine that's already set up.

The `<hostname>-` prefix keeps machines separate when the same workspace
is used on more than one host, so `laptop-...` and `workstation-...`
pools never collide even though both encode the same `~/OPS` cwd.

Stage 2 (`.claude-config/deploy.{sh,ps1}`) runs
`ac-memory-init --auto-commit` during deploy, so a new machine wires
this automatically and commits its new pool. See `DEPLOYMENT.md`.

**Second profile.** If you run a second Claude Code profile via
`CLAUDE_CONFIG_DIR`, `ac-memory-init` points that profile's per-cwd
memory symlinks at the SAME shared pool here — auto-memory is identical
across profiles rather than forked.

## What's inside each pool

Auto-memory entries follow the format documented in the user-level
`CLAUDE.md` (auto-memory section). Each entry is a single `.md` file
with frontmatter:

```markdown
---
name: <short-kebab-case-slug>
description: <one-line summary>
metadata:
  type: <user | feedback | project | reference>
---

<memory body>
```

Each pool also holds a `MEMORY.md` index — one line per memory file.
`MEMORY.md` is regenerated by `WORKFORCE/bin/ac-memory-index` and is
loaded into every session automatically; the individual entry files are
loaded on demand when their `description` matches the work at hand.

## Housekeeping tools

- **`ac-memory-init`** — initialize (or re-sync) all of this machine's
  memory pools. Run once per machine; the deployer runs it for you.
- **`ac-memory-index`** — regenerate a pool's `MEMORY.md` index after
  entries are added or removed.
- **`ac-memory-gc`** — read-only inventory + GC report. Flags each pool
  as active / stale / empty / over-limit and STAGES prune and archive
  candidates for you to act on; it never deletes or moves anything on
  its own (deletes stay operator-gated per the never-delete-without-
  approval rule). The `ops-memory-gc` systemd timer runs it weekly and
  drops the report at `~/.local/state/ops/memory-gc-last.txt`.
- **`memory-prune` skill** — the AI-driven counterpart to `ac-memory-gc`:
  fans a classification pass over every entry and returns a
  keep / promote / move / discard action table, then executes it after
  you sign off. Use it for the periodic audit; use `ac-memory-gc` for
  the fast mechanical inventory.

## What goes here vs `CONTEXT/`

- `.claude-memory/` — durable Claude-written context built up by AI
  sessions over time. Auto-managed; sessions add, update, and (with
  approval) remove entries as they learn.
- `CONTEXT/` — Operator-authored identity + doctrine files. Edited by
  the Operator (or by an AI with the Operator's approval), not
  auto-managed.

A memory entry that has hardened into durable identity or doctrine
should be promoted OUT of here and INTO `CONTEXT/` — the `memory-prune`
skill surfaces those promotion candidates.

## What to read on session resume

On compaction or a fresh session, Claude Code automatically loads the
`MEMORY.md` for the current workspace. Sessions don't need to manually
`Read` this folder — entries are surfaced by the memory loader. The only
reason to look here directly is to debug memory drift or hand-edit an
outdated entry.

## Git note

This directory is intentionally NOT gitignored, even though it sits next
to `~/.claude/` machinery. Tracking it across machines is the feature —
don't fold it into `.gitignore`.
